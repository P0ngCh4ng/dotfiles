# 実装計画: UI生成・修正エージェントシステム（階層的エージェント構成）

**作成日**: 2026-03-03
**ステータス**: 計画承認待ち
**アプローチ**: 階層的エージェントシステム + Playwright MCP統合

---

## 概要

Claude Code開発時に、ペルソナと要件に基づいてUIをゼロから生成し、**Playwright MCP（agent-browser）を使って実際のブラウザで動作確認しながら**アクセシビリティ・レスポンシブ・レイアウト品質を包括的にチェック・修正する階層的エージェントシステムを構築します。

---

## 要件定義

### 問題定義
Claude Codeでの開発時に、UIをゼロから作成したり、既存UIの品質を改善したりする作業を、要件とペルソナに基づいて自動化し、一貫性のある高品質なUIを効率的に生成・改善したい。

### 主要要件

#### 1. UI生成エージェント
- **要件ベースの生成**: 既存の要件や機能を満たすUIを生成（要件が不明確な場合は別エージェントで明確化）
- **ペルソナ駆動設計**: ペルソナを作成し、そのペルソナが扱いやすいUIを設計
  - ペルソナ定義の有無を確認 → なければ推測 or 対話的に作成
  - プロジェクト内に保存して再利用可能に
- **実在デザインの参照**: 完全オリジナルではなく、Web検索で実在するデザインパターンを調査・参考にする
- **技術スタック自動検出**: プロジェクトのReact/Vue/HTML等を検出、不可能なら選定から実施
- **スタイリング方法の適応**: CSS Modules/Tailwind/styled-components等をプロジェクトに合わせる

#### 2. UI修正エージェント（サブエージェント構成）
以下のステップで構成:
- **振り分けステップ**: チェック項目を分類・優先順位付け
- **ブラウザ起動**: Playwright MCPでローカル開発サーバーのUIを開く
- **チェックステップ**: 各項目を並列的にチェック
  - アクセシビリティ（ARIA、キーボード操作、色コントラスト）
  - レスポンシブ対応（モバイル/タブレット/デスクトップ）
  - レイアウト品質（要素の重なり、サイズ、配置）
  - UIの一貫性（色、フォント、間隔等）
- **修正可否判断ステップ**: 問題を「修正可能」「要相談」等に分類
- **修正実行ステップ**: 承認された修正を実施

#### 3. 判断エージェント
- 修正エージェントの各ステップ間で自動的に判断を下す
- 判断の根拠やロジックをログとして記録
- 判断困難なケース（デザインの好み、大規模変更等）のみユーザーに確認
- 判断基準は要件やプロジェクト文脈から推測

#### 4. Playwright MCP統合（agent-browser）
- **実ブラウザでの検証**: Playwright MCPツール（`mcp__playwright__*`）を使用
- **アクセシビリティスナップショット**: `browser_snapshot` で構造化データ取得
- **レスポンシブテスト**: `browser_resize` で複数ビューポート検証
- **視覚的検証**: `browser_take_screenshot` でスクリーンショット取得
- **インタラクション検証**: `browser_click`, `browser_type` 等でUIの動作確認

### ワークフロー

```
[UI生成要求]
    ↓
UI生成エージェント起動
    ↓
1. ペルソナ確認/作成
2. 要件確認
3. デザイン調査（Web検索）
4. 技術スタック検出
5. UI生成
    ↓
生成結果をユーザーに提示
    ↓
ユーザーが修正エージェント起動を判断
    ↓
[修正エージェント起動]
    ↓
振り分け → ブラウザ起動 → 判断エージェント → チェック（並列） → 判断エージェント → 修正可否判断 → 判断エージェント → 修正実行
    ↓
修正結果をユーザーに提示して承認
    ↓
確定
```

### 起動方法（3つの方法すべて対応）

#### 1. スラッシュコマンド
- `/ui-generate`: UI生成エージェント単独起動
- `/ui-review`: UI修正エージェント単独起動
- `/ui`: 統括コマンド（生成→修正の完全フロー）

#### 2. 自動検知
- 「UIを作って」→ UI生成エージェント起動
- 「UIをレビューして」→ UI修正エージェント起動

#### 3. 他エージェントからの呼び出し
- plannerやcode-reviewer等から必要に応じて呼び出し可能

## 実装ステップ

### Phase 1: コアエージェント定義（3ファイル）

#### 1. **統括エージェント作成** (`~/.claude/agents/ui-orchestrator.md`)
- **アクション**: UI生成・修正の全体フローを制御するエージェント定義
- **なぜ**: 生成→ユーザー確認→修正のワークフロー管理
- **依存**: なし
- **リスク**: 低
- **ツール**: `Task`, `Read`, `Grep`, `Glob`
- **モデル**: sonnet

#### 2. **UI生成エージェント作成** (`~/.claude/agents/ui-generator.md`)
- **アクション**: ペルソナ確認/作成、要件確認、デザイン調査、技術スタック検出、UI生成を実行
- **なぜ**: ゼロからのUI生成ロジック
- **依存**: なし
- **リスク**: 中 - Web検索の品質、技術スタック検出の精度
- **ツール**: `Read`, `Write`, `Edit`, `Grep`, `Glob`, `WebSearch`, `Bash`
- **モデル**: sonnet

#### 3. **判断エージェント作成** (`~/.claude/agents/ui-decision-maker.md`)
- **アクション**: 修正エージェントの各ステップ間で自動判断、ログ記録、困難時ユーザーエスカレーション
- **なぜ**: 修正フローの自動化と判断品質の一貫性
- **依存**: なし
- **リスク**: 中 - 判断ロジックの妥当性
- **ツール**: `Read`, `Grep`, `Glob`, `Write` (ログ記録用)
- **モデル**: sonnet

### Phase 2: 修正エージェントシステム + Playwright MCP統合（5ファイル）

#### 4. **UI修正エージェント作成** (`~/.claude/agents/ui-reviewer.md`)
- **アクション**: 振り分け→**ブラウザ起動**→チェック→修正可否判断→修正実行のワークフロー制御
- **なぜ**: 実ブラウザでの検証を含む修正プロセス全体の管理
- **依存**: ステップ3（判断エージェント）
- **リスク**: 中 - Playwright MCP環境の設定要
- **ツール**: `Task`, `Read`, `Grep`, `Glob`, `Bash`, `mcp__playwright__browser_navigate`, `mcp__playwright__browser_close`, `mcp__playwright__browser_install`
- **モデル**: sonnet

#### 5. **アクセシビリティチェッカー作成** (`~/.claude/agents/ui-accessibility-checker.md`)
- **アクション**:
  - `browser_snapshot` でアクセシビリティツリー取得
  - ARIA属性、role、セマンティックHTMLを検証
  - `browser_click`, `browser_press_key` でキーボード操作テスト
  - 色コントラストを検証（スクリーンショット + 解析）
- **なぜ**: WCAG準拠の実ブラウザベース検証
- **依存**: なし
- **リスク**: 低
- **ツール**: `Read`, `Grep`, `Glob`, `mcp__playwright__browser_snapshot`, `mcp__playwright__browser_navigate`, `mcp__playwright__browser_click`, `mcp__playwright__browser_press_key`, `mcp__playwright__browser_take_screenshot`
- **モデル**: sonnet

#### 6. **レスポンシブチェッカー作成** (`~/.claude/agents/ui-responsive-checker.md`)
- **アクション**:
  - `browser_resize` で複数ビューポート（320px, 768px, 1024px, 1920px）に変更
  - 各サイズで `browser_snapshot` + `browser_take_screenshot` で検証
  - レイアウト崩れ、オーバーフロー、タッチ領域を確認
- **なぜ**: 実デバイスサイズでのマルチデバイス対応検証
- **依存**: なし
- **リスク**: 低
- **ツール**: `Read`, `Grep`, `Glob`, `mcp__playwright__browser_resize`, `mcp__playwright__browser_take_screenshot`, `mcp__playwright__browser_snapshot`, `mcp__playwright__browser_navigate`
- **モデル**: sonnet

#### 7. **レイアウトチェッカー作成** (`~/.claude/agents/ui-layout-checker.md`)
- **アクション**:
  - `browser_snapshot` で要素の位置・サイズを取得
  - `browser_take_screenshot` で視覚的レイアウト確認
  - 要素の重なり、z-index、ボタンタップ領域（最小44x44px）を検証
  - `browser_click` でクリック可能性テスト
- **なぜ**: 実ブラウザでのUIの物理的品質保証
- **依存**: なし
- **リスク**: 低
- **ツール**: `Read`, `Grep`, `Glob`, `mcp__playwright__browser_snapshot`, `mcp__playwright__browser_take_screenshot`, `mcp__playwright__browser_click`, `mcp__playwright__browser_navigate`
- **モデル**: sonnet

#### 8. **UI一貫性チェッカー作成** (`~/.claude/agents/ui-consistency-checker.md`)
- **アクション**:
  - `browser_take_screenshot` で複数ページのスクリーンショット取得
  - 色、フォント、間隔、ボタンスタイルを比較
  - プロジェクトのデザインシステム（Tailwind config, CSS変数等）と照合
- **なぜ**: ブランド・デザイン一貫性の視覚的検証
- **依存**: なし
- **リスク**: 中 - デザインシステム定義の検出
- **ツール**: `Read`, `Grep`, `Glob`, `mcp__playwright__browser_take_screenshot`, `mcp__playwright__browser_snapshot`, `mcp__playwright__browser_navigate`
- **モデル**: sonnet

## テスト戦略

### 単体テスト（エージェント個別）
- 各チェッカーエージェントがPlaywright MCPツールを正しく使えるか
- `browser_snapshot`, `browser_resize`, `browser_take_screenshot` が正常動作するか
- 判断エージェントが適切な判断を下せるか
- ペルソナ管理が正しくJSONを読み書きできるか

### 統合テスト（フロー全体）
- `/ui` コマンドで生成→**ブラウザ起動**→修正の完全フローが動作するか
- 複数ビューポートでのレスポンシブチェックが並列実行できるか
- ブラウザの起動・終了が正しく管理されるか
- 自動検知（「UIを作って」）が正しくエージェントを起動するか
- ユーザー確認ポイントで正しく中断・再開できるか

### E2Eテスト（実プロジェクト + 実ブラウザ）
- React + Tailwindプロジェクトで実ブラウザ検証
- Vue + CSS Modulesプロジェクトで実ブラウザ検証
- 技術スタック検出失敗時の選定フローの動作確認
- Playwright MCPが利用不可の場合のフォールバック動作確認

## リスクと緩和策

### **リスク**: Playwright MCP環境が未設定またはブラウザ未インストール
- **緩和策**: `mcp__playwright__browser_install` を修正エージェント起動時に自動実行、失敗時は静的解析にフォールバック

### **リスク**: ローカル開発サーバーが起動していない（ブラウザで開けない）
- **緩和策**: 修正エージェントがサーバー起動コマンド（`npm run dev`, `yarn dev`等）を検出・実行、URLを自動推測（localhost:3000等）

### **リスク**: ブラウザ操作の並列実行でリソース競合
- **緩和策**: チェッカーサブエージェント間でブラウザタブを分離（`browser_tabs` ツール活用）

### **リスク**: Web検索によるデザイン参照の品質が不安定
- **緩和策**: デザインパターンのフォールバック（Material Design、Ant Design等）をエージェントに組み込む

### **リスク**: 技術スタック検出の精度（package.json不在、複数フレームワーク混在等）
- **緩和策**: 検出失敗時に対話的にユーザーに確認する仕組み

### **リスク**: 判断エージェントの判断ロジックが曖昧
- **緩和策**: 判断基準をエージェント定義内に明記、ログを詳細に記録してユーザーが追跡可能に

### **リスク**: ペルソナ管理がプロジェクトローカルのため、グローバル再利用不可
- **緩和策**: Phase 4後にグローバルペルソナ管理機能を追加（将来拡張）
