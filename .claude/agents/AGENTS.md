# Agent Registry

このファイルは`~/.claude/agents/`ディレクトリ内のエージェント定義を管理します。

**重要**: エージェントを追加・削除した際は、必ずこのファイルを更新してください。

## 利用可能なエージェント

### code-reviewer

**用途**: コード品質・セキュリティ・保守性の専門的レビュー

**いつ使うか**:
- コードを書いた後、すぐにレビューが必要な時
- セキュリティ脆弱性をチェックしたい時
- コード品質を向上させたい時
- ベストプラクティスに準拠しているか確認したい時

**自動起動トリガー**:
- コードの記述・修正完了直後（重要なコード変更後は必須）
- ユーザーが「レビューして」「品質チェック」等を要求した時

**モデル**: sonnet

### design-review-agent

**用途**: フロントエンドPRやUI変更の包括的デザインレビュー

**いつ使うか**:
- UI/UXコンポーネントを変更した時
- 視覚的一貫性を検証したい時
- アクセシビリティをチェックしたい時
- レスポンシブデザインを確認したい時

**自動起動トリガー**:
- PRでUIコンポーネント、スタイル、ユーザー向け機能を変更した時
- ユーザーが「デザインレビュー」「UI確認」等を要求した時

**モデル**: sonnet

### doc-updater

**用途**: ドキュメントとコードマップの専門更新

**いつ使うか**:
- コードマップを更新したい時
- READMEやガイドを更新したい時
- ドキュメントを最新の実装に同期させたい時
- `/update-codemaps`や`/update-docs`を実行したい時

**自動起動トリガー**:
- ユーザーが「ドキュメント更新」「コードマップ更新」等を要求した時
- 大規模なコード変更後にドキュメント同期が必要な時

**モデル**: sonnet

### implementation-bridge

**用途**: 要件が明確だが実装パスの構造化が必要な時の支援

**いつ使うか**:
- 要件は明確だが、実装手順が不明確な時
- 依存関係を整理したい時
- 技術的アプローチの提案が欲しい時
- 実装のギャップを特定したい時

**自動起動トリガー**:
- ユーザーが「どう実装すべきか」「実装手順」等を要求した時
- 要件は明確だが実装方法が不明な場合

**モデル**: sonnet

### interviewer

**用途**: 要件明確化(WHAT)と実装構造化(HOW)の適切な専門家への委譲ルーター

**いつ使うか**:
- ユーザーが何を作りたいかが不明確な時
- 実装の構造化が必要な時
- 適切な専門家エージェントへの委譲判断が必要な時

**自動起動トリガー**:
- 曖昧な要求や複雑な実装相談を受けた時
- requirements-interviewerまたはimplementation-bridgeの判断が必要な時

**モデル**: sonnet

### planer

**用途**: 複雑な機能・リファクタリングの実装計画の専門立案

**いつ使うか**:
- 複雑な機能の実装計画が必要な時
- アーキテクチャ変更の計画を立てたい時
- 複雑なリファクタリングの計画が必要な時
- 段階的な実装戦略が欲しい時

**自動起動トリガー**:
- ユーザーが機能実装、アーキテクチャ変更、複雑なリファクタリングを要求した時
- プロアクティブに計画タスクに使用

**モデル**: sonnet

### requirements-interviewer

**用途**: WHAT（何を作るか）を明確化するためのソクラテス式質問専門家

**いつ使うか**:
- ユーザーの要求が曖昧な時
- 問題、ユーザー、ゴール、成功基準を明確化したい時
- 構造的対話で要件を引き出したい時

**自動起動トリガー**:
- ユーザーが曖昧な要求をした時（「〜を改善したい」「もっと良くしたい」等）
- 要件明確化が必要な新機能の相談を受けた時

**モデル**: sonnet

## UI生成・レビューシステム

### ui-orchestrator

**用途**: ペルソナ駆動UI生成と包括的品質レビューの完全ワークフロー管理

**いつ使うか**:
- ゼロからUIを生成したい時
- 既存UIの品質を改善したい時
- ペルソナベースのUI設計をしたい時
- 生成→レビューの完全フローを実行したい時

**自動起動トリガー**:
- ユーザーが「UIを作って」「UIをレビューして」「UIを改善して」等を要求した時
- 「ペルソナベースでUI」「アクセシブルなUI」「レスポンシブなUI」等のキーワード

**利用可能なツール**: Task, Read, Grep, Glob, WebSearch

**モデル**: sonnet

### ui-generator

**用途**: ペルソナと要件に基づくUI生成（デザインパターン調査含む）

**いつ使うか**:
- 新しいUIコンポーネントを作成したい時
- ペルソナに合わせたUIが必要な時
- 実在のデザインパターンを参考にしたい時
- 技術スタックに適したUIコードが欲しい時

**自動起動トリガー**:
- `/ui-generate` コマンド実行時
- ui-orchestratorから生成フェーズで呼び出される時

**利用可能なツール**: Read, Write, Edit, Grep, Glob, WebSearch, Bash, Task

**モデル**: sonnet

### ui-reviewer

**用途**: Playwright MCPを使った実ブラウザでの包括的UI品質レビュー

**いつ使うか**:
- UIの品質を包括的にチェックしたい時
- アクセシビリティ、レスポンシブ、レイアウト、一貫性を確認したい時
- 実際のブラウザでの動作を検証したい時
- 複数ビューポートでのテストが必要な時

**自動起動トリガー**:
- `/ui-review` コマンド実行時
- ui-orchestratorからレビューフェーズで呼び出される時
- design-review-agentと併用可能

**利用可能なツール**: Task, Read, Grep, Glob, Bash, mcp__playwright__browser_*

**モデル**: sonnet

### ui-decision-maker

**用途**: UIレビュー結果の自動判断と修正優先順位付け

**いつ使うか**:
- レビュー結果の自動分類が必要な時
- 自動修正可能な問題を判定したい時
- 修正の優先順位を決めたい時
- ユーザー相談が必要な問題を識別したい時

**自動起動トリガー**:
- ui-reviewerから各ステップ間で呼び出される時
- 判断困難時のエスカレーションが必要な時

**利用可能なツール**: Read, Grep, Glob, Write

**モデル**: sonnet

### ui-accessibility-checker

**用途**: WCAG 2.1 AA準拠の実ブラウザアクセシビリティ検証

**いつ使うか**:
- アクセシビリティをテストしたい時
- WCAG準拠を確認したい時
- キーボードナビゲーションを検証したい時
- 色コントラストを測定したい時

**自動起動トリガー**:
- ui-reviewerから並列チェックの一部として呼び出される時

**利用可能なツール**: Read, Grep, Glob, mcp__playwright__browser_*

**モデル**: sonnet

### ui-responsive-checker

**用途**: モバイル/タブレット/デスクトップでのレスポンシブデザイン検証

**いつ使うか**:
- 複数デバイスでの表示を確認したい時
- タッチターゲットサイズを検証したい時
- レイアウト崩れをチェックしたい時
- ブレークポイント動作を確認したい時

**自動起動トリガー**:
- ui-reviewerから並列チェックの一部として呼び出される時

**利用可能なツール**: Read, Grep, Glob, mcp__playwright__browser_*

**モデル**: sonnet

### ui-layout-checker

**用途**: 要素配置、間隔、重なり、視覚的階層の品質検証

**いつ使うか**:
- 要素の重なりをチェックしたい時
- 間隔の一貫性を確認したい時
- クリック可能領域を検証したい時
- 視覚的階層を確認したい時

**自動起動トリガー**:
- ui-reviewerから並列チェックの一部として呼び出される時

**利用可能なツール**: Read, Grep, Glob, mcp__playwright__browser_*

**モデル**: sonnet

### ui-consistency-checker

**用途**: カラー、タイポグラフィ、コンポーネントスタイルのデザイン一貫性検証

**いつ使うか**:
- デザインシステム準拠を確認したい時
- 色の一貫性をチェックしたい時
- タイポグラフィの統一性を検証したい時
- 複数ページ間の一貫性を確認したい時

**自動起動トリガー**:
- ui-reviewerから並列チェックの一部として呼び出される時

**利用可能なツール**: Read, Grep, Glob, mcp__playwright__browser_*

**モデル**: sonnet

## エージェント自動起動ルール

以下の条件でエージェントを**プロンプト不要で自動起動**すること:

### 即座に起動すべきケース

1. **コードレビュー** → `code-reviewer`
   - 重要なコード変更後は必須
   - ユーザーが「レビューして」「品質チェック」等を要求

2. **複雑な機能・リファクタリング** → `planer`
   - 機能実装、アーキテクチャ変更の要求
   - 複雑なリファクタリングの計画

3. **曖昧な要求** → `interviewer` または `requirements-interviewer`
   - ユーザーの要求が不明確な時
   - 要件明確化が必要な時

4. **ドキュメント更新** → `doc-updater`
   - 大規模なコード変更後
   - ユーザーが明示的にドキュメント更新を要求

## エージェント作成・削除時の必須作業

**エージェントを追加した場合**:
1. `~/.claude/agents/AGENTS.md` (このファイル)を更新
2. エージェント名、用途、自動起動トリガーを記載
3. 利用可能なツールとモデルを明記

**エージェントを削除した場合**:
1. `~/.claude/agents/AGENTS.md` から該当エージェントのセクションを削除
2. 削除理由をメモとして記録

## エージェント定義フォーマット

新しいエージェントは以下のフォーマットで追加:

```markdown
### agent-name

**用途**: エージェントの主要目的を1行で説明

**いつ使うか**:
- 具体的なユースケース1
- 具体的なユースケース2
- 具体的なユースケース3

**自動起動トリガー**:
- トリガー条件1
- トリガー条件2

**利用可能なツール**: ツールのリスト

**モデル**: sonnet/haiku/opus
```
