# ONLINE MEDIC Architecture Skill

## Overview
Complete architectural understanding of ONLINE MEDIC - an automated online medical consultation system via LINE Bot that integrates Google Forms, Calendar, Meets, and Stripe payments.

## Core Components

### 1. Entry Points
- **LINE Bot Webhook**: `/callback` endpoint receives all LINE events
- **Friend Add**: Triggers initial flow with booking code generation
- **Postback Actions**: `action=online_consul` starts consultation flow

### 2. Key Use Cases
```
ReminderUseCase  - Form sync, Calendar monitoring, reminder notifications
PaymentUseCase   - Stripe payment processing
LineUseCase      - LINE message sending utilities
ChatController   - Webhook event routing
```

### 3. External Integrations
- **Google Forms API**: Patient questionnaire submission
- **Google Calendar API**: Appointment scheduling (read-only)
- **Google Meets**: Video consultation URLs (auto-generated by Calendar)
- **Stripe API**: Payment processing
- **LINE Messaging API**: Push notifications

## Critical Technical Details

### Booking Code System (CL-XXXXX)
**Purpose**: Bridge Google Form submissions with LINE user IDs
**Format**: `CL-` + random 5-character alphanumeric (uppercase)
**Storage**: `BookingCode` table
**Flow**:
1. Generated on friend add or online consultation request
2. Prefilled in Google Form URL
3. Used to lookup LINE ID when form is submitted

### Email ↔ LINE ID Mapping
**Purpose**: Reverse lookup from Google Calendar attendee email to LINE ID
**Storage**: `UserEmail` table
**Critical**: Without this mapping, reminders cannot be sent
**Process**:
1. User submits form with email
2. System creates email → line_id mapping
3. Calendar attendee email → find line_id → send LINE reminder

### Reminder Deduplication
**Mechanism**: Store sent reminders in `Reminder` table
**Check**: Same `(line_id, event_id, reminder_type)` tuple sent only once
**Prevents**: Duplicate notifications if cron runs multiple times

### Cron Jobs (Every 5 Minutes)
```typescript
ReminderCronService.start()
  ├─ syncFormResponses()      // Sync new form submissions
  └─ checkAndSendReminders()  // Check calendar & send reminders
```

## Key Implementation Patterns

### 1. Booking Code Generation
```typescript
generateBookingCode(line_id: string): Promise<string>
  ├─ Check if existing code exists for user
  ├─ If exists: return existing code
  ├─ If not: generate new CL-XXXXX code
  └─ Save to BookingCode table
```

### 2. Form URL with Prefill
```typescript
getGoogleFormUrl(bookingCode: string): string
  ├─ Base URL: process.env.GOOGLE_FORM_URL
  ├─ Entry ID: process.env.GOOGLE_FORM_BOOKING_CODE_ENTRY
  └─ Return: {formUrl}?{entryId}={bookingCode}
```

### 3. Calendar Event Monitoring
```typescript
checkAndSendReminders()
  ├─ Fetch events 0-70 minutes ahead
  ├─ For each event:
  │   ├─ Get attendee emails
  │   ├─ Convert email → line_id (via UserEmail table)
  │   ├─ Calculate minutes until start
  │   └─ Send appropriate reminder:
  │       ├─ 55-65min: 60-minute reminder
  │       ├─ 25-35min: 30-minute reminder
  │       └─ 0-25min: Immediate reminder
  └─ Record sent reminder (prevent duplicates)
```

### 4. Reminder Message Construction
```typescript
sendReminderNotification(line_id, startTime, reminderType, meetLink)
  ├─ Format time in Asia/Tokyo timezone
  ├─ Select timing text based on reminderType
  ├─ Include Google Meets or LIFF URL
  └─ Push message via LINE Client
```

## Current Limitations

1. **Manual Calendar Registration**:
   - Patient must manually book appointment via Calendar link
   - No automatic calendar event creation

2. **Email Required**:
   - Google Form must collect email
   - Email is critical for Calendar attendee → LINE ID mapping

3. **Calendar Sharing**:
   - Service Account needs read access to Google Calendar
   - Must be configured in Google Calendar settings

4. **Form Sync Delay**:
   - Runs every 5 minutes via cron
   - Not real-time (acceptable for this use case)

### Scenario 3: Reminder sent
```

  → ReminderCronService.checkReminders()
  → ReminderUseCase.checkAndSendReminders()
  → Fetch Calendar events (0-70min ahead)
  → For each event:
      ├─ Get attendee email
      ├─ Lookup email → line_id
      ├─ Calculate minutes until start
      ├─ Check if already sent (Reminder table)
      └─ Send LINE push message with Meets URL
```

## Architecture Principles

1. **Event-Driven**: Cron jobs drive form sync and reminder sending
2. **Idempotent**: Duplicate protection via Reminder table
3. **Cross-Platform Mapping**: Booking codes and email mappings bridge ecosystems
4. **Separation of Concerns**: Each UseCase handles one domain (Reminders, Payments, LINE UI)
5. **External Integration**: Heavy reliance on Google APIs and Stripe
6. **Minimal Database**: Only 6 models needed for core functionality
